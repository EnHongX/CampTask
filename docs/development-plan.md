# 开发计划

## 该不该现在写计划

该做。

## 为什么

当前已经确定 4 个端、3 个后端服务、Node.js 技术栈和主业务流程。继续直接写代码会导致范围失控，先固化计划可以控制 MVP。

## 阶段 0：基础设计

目标：把开工前必须统一的规则定住。

交付物：

- 数据模型草案：`docs/data-model.md`
- 权限模型：`docs/permission-model.md`
- API 规范：`docs/api-guidelines.md`
- 本地启动方式：`docs/local-development.md`

验收标准：

- 明确核心表。
- 明确角色和数据范围。
- 明确 API 路径、响应、分页、错误码。
- 明确本地不用 Docker Compose。

## 阶段 1：项目脚手架

目标：项目能本地启动。

后端：

- 初始化 `pnpm workspace`。
- 初始化 `identity-service`，端口 `5301`。
- 初始化 `camp-service`，端口 `5302`。
- 初始化 `notification-service`，端口 `5303`。
- 配置 `TypeScript`、`NestJS`、`Prisma`。
- 增加健康检查接口。

前端：

- 初始化 `student-web`。
- 初始化 `assistant-web`。
- 初始化 `instructor-web`。
- 初始化 `admin-web`。
- 配置 `Vue 3`、`Vite`、`Pinia`、`Element Plus`。

验收标准：

- 3 个后端服务可本地启动。
- 4 个前端应用可本地启动。
- 每个后端服务有 `/health`。

## 阶段 2：管理后台基础

目标：先把业务基础数据建起来。

范围：

- 课程管理。
- 班级管理。
- 用户角色分配。
- 时间规则配置。
- 评分配置管理。

验收标准：

- 管理员可以创建课程。
- 管理员可以创建班级并关联课程。
- 管理员可以给用户分配角色。
- 管理员可以维护基础时间规则和评分配置。

## 阶段 3：作业主流程

目标：打通最小闭环。

范围：

- 讲师发布作业。
- 学员查看任务。
- 学员提交作业。
- 助教查看待处理任务。
- 助教批改、评论、评分、退回。
- 学员查看反馈。
- 学员重提。

验收标准：

```text
讲师发布任务 -> 学员提交 -> 助教批改/退回 -> 学员重提 -> 助教再次批改
```

## 阶段 4：讲师统计

目标：让讲师能判断班级作业完成情况。

范围：

- 班级完成情况。
- 提交率。
- 逾期率。
- 平均分。
- 成绩分布。

验收标准：

- 讲师可以查看自己负责班级的作业统计。
- 统计默认按最新有效提交版本计算。

## 阶段 5：通知提醒

目标：补齐提醒能力，但不阻塞主流程。

范围：

- 作业发布提醒。
- 截止提醒。
- 批改完成提醒。

验收标准：

- 第一版可以先写入站内通知或通知日志。
- 不强制接入短信、邮件、企业微信。

## 阶段 6：上线准备

目标：支持小范围真实试用。

范围：

- 表单校验。
- 权限测试。
- 状态流转测试。
- 错误处理。
- 运行日志。
- 数据库迁移。
- 初始化管理员账号。
- 部署说明。

## 暂不做

- ⚠️ 技术过度设计：不做微服务治理。
- ⚠️ 时间黑洞：不做复杂规则引擎。
- ⚠️ 收益不匹配：不做复杂报表中心。
- ⚠️ 伪需求：不做自动评分、查重、申诉复核。

# 服务划分

## 该不该拆成多个独立服务

第一阶段不该把课程、作业、提交、批改、规则、统计拆成多个独立运行服务。

## 为什么

这些能力强依赖同一条作业生命周期，早拆会带来跨服务事务、状态同步、联调和部署成本。

当前更合理的做法是保留领域模块边界，但运行单元先收敛成 3 个服务。

## 服务清单

| 服务 | 端口 | 职责 |
| --- | --- | --- |
| 身份与权限服务 | `5301` | 用户、角色、登录态、权限判断。 |
| 训练营业务服务 | `5302` | 课程、班级、作业、提交、批改、规则、统计。 |
| 通知提醒服务 | `5303` | 发布提醒、截止提醒、批改完成提醒、通知模板。 |

## 领域模块

`camp-service` 内部按领域模块组织：

- 课程与班级模块。
- 作业模块。
- 提交模块。
- 批改与反馈模块。
- 截止时间与规则模块。
- 统计模块。

## 实施建议

第一版使用单仓库多服务结构。等业务稳定、团队规模或部署诉求明确后，再决定是否把 `camp-service` 内部模块拆成独立进程或独立仓库。

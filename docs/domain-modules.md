# 领域模块

## 该不该独立成服务

第一阶段不独立成服务，先作为 `camp-service` 内部模块。

## 为什么

课程、作业、提交、批改、规则、统计都围绕同一条作业生命周期，数据关系密集。早期拆服务会让简单流程变成跨服务调用。

## 模块清单

| 模块 | 职责 | 所属服务 |
| --- | --- | --- |
| 课程与班级模块 | 课程、班级、讲师分配、学员归属 | `camp-service` |
| 作业模块 | 作业发布、编辑、截止时间、附件要求 | `camp-service` |
| 提交模块 | 文本、链接、文件引用、提交版本、重提 | `camp-service` |
| 批改与反馈模块 | 评分、评论、退回、反馈历史 | `camp-service` |
| 截止时间与规则模块 | 截止时间、补交规则、重提规则、评分规则引用 | `camp-service` |
| 统计模块 | 提交率、逾期率、平均分、成绩分布、班级完成情况 | `camp-service` |

## 后续拆分条件

满足以下条件之一，再考虑从 `camp-service` 拆出独立服务：

- 单个模块有独立部署、扩容或稳定性诉求。
- 团队已经按模块分工，代码冲突明显增加。
- 模块数据边界稳定，跨模块事务很少。
- 外部系统需要单独集成某个能力。

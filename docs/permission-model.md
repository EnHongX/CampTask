# 权限模型

## 该不该做复杂权限系统

不该。

## 为什么

CampTask 第一版只需要覆盖 4 类角色和班级/课程数据范围。复杂权限表达式、组织树和策略引擎会增加实现成本，但不能直接提升 MVP 验证效率。

## 权限方案

第一版使用 `RBAC + 数据范围`。

### 角色

- `STUDENT`：学员。
- `ASSISTANT`：助教。
- `INSTRUCTOR`：讲师。
- `ADMIN`：管理员。

一个用户可以有多个系统角色。

### 数据范围

数据范围由用户和课程/班级关系决定。

- 学员：只能访问自己所在班级的作业和自己的提交。
- 助教：只能访问被分配班级的待批改提交。
- 讲师：只能访问自己负责课程/班级的作业和统计。
- 管理员：可以维护全局课程、班级、角色、时间规则、评分配置。

## 端权限

### 学员端

允许：

- 查看自己班级内已发布任务。
- 提交自己的作业。
- 查看自己的反馈。
- 对被退回的作业重提。

禁止：

- 查看其他学员提交。
- 批改作业。
- 发布作业。
- 修改课程和班级配置。

### 助教端

允许：

- 查看分配范围内的待处理任务。
- 批改。
- 评论。
- 退回。
- 评分。

禁止：

- 发布作业。
- 修改课程和班级配置。
- 查看未分配班级的数据。

### 讲师端

允许：

- 在负责范围内发布作业。
- 查看班级完成情况。
- 查看成绩与统计。

禁止：

- 修改全局用户角色。
- 修改非负责课程/班级的数据。

### 管理后台

允许：

- 课程管理。
- 班级管理。
- 角色分配。
- 时间规则配置。
- 评分配置管理。

禁止：

- 越过业务规则直接改提交状态。
- 代替助教批改，除非后续明确管理员代操作需求。

## 后端校验原则

- 前端隐藏菜单不是权限控制。
- 所有接口必须在后端校验角色。
- 所有涉及课程、班级、作业、提交的数据接口必须校验数据范围。
- 修改接口必须校验资源当前状态。

## 接口鉴权建议

- 第一版使用登录态或 `JWT` 均可。
- API 请求统一携带身份凭证。
- 后端解析出 `currentUser`。
- 业务接口基于 `currentUser` 做角色和数据范围判断。

## 暂不做

- ⚠️ 技术过度设计：不做动态权限表达式。
- ⚠️ 技术过度设计：不做策略引擎。
- ⚠️ 时间黑洞：不做复杂组织树权限。
